# 8250 Tier C: serial8250_handle_irq_locked — landed, live-wired — 2026-07-18

Status: **achieved**, the third Tier C slice for
[awto-au/linux-rs#25](https://github.com/awto-au/linux-rs/issues/25). Issue
#25 stays **open** (`set_termios` and `autoconfig` remain, each its own
future scope). Kernel commit: `600a75584748` on `linux-rs/phase2-gcd`,
pushed to `awtoau/linux`. TU 39.

This slice follows the same day's
`docs/8250-tier-c-startup-shutdown-2026-07-18.md` (`serial8250_do_startup`/
`serial8250_do_shutdown`, TU 38) directly, reusing its translation pattern
(opaque `*mut c_void` port + narrow extern "C" shims) and its KUnit-exception
precedent, but is **explicitly treated as higher risk**, per
`docs/8250-tier-c-blocker-2026-07-18.md`'s own evidence-based ranking:
`serial8250_handle_irq_locked` runs on *every* live interrupt (not just
device bring-up/teardown), asserts `lockdep_assert_held_once(&port->lock)`
on the real port lock, and has zero test margin — a bug here is immediately
live on the console path the moment it's wired, with no graceful
degradation. Per the task's own instruction, this is not treated as a
routine follow-up: the verification bar below uses *more* repeat boots (8
total, vs. the prior slice's 6) than the already-elevated bar the prior
slice used.

## What was translated

`serial8250_handle_irq_locked()` — the live IRQ RX/TX dispatch control flow
(branching, condition ordering; the function has no early returns) — to
`drivers/tty/serial/8250/8250_irq_rs.rs`. Every branch and condition matches
the C original line-for-line (verified by direct side-by-side reading
against the `#else` arm, which keeps the original C body verbatim for
exactly this comparison, same discipline as the prior slice).

### Callees stay in C, exactly as scoped

`serial8250_clear_and_reinit_fifos`, `serial8250_rx_chars`,
`serial8250_modem_status`, `serial8250_tx_chars`, `__stop_tx`, and
`handle_rx_dma` are **not** translated by this TU (confirmed via grep before
starting: none of this project's Rust files reference any of them). They
remain unmodified C functions, called through the shim boundary described
below, with call signatures and argument marshaling preserved exactly.
`handle_rx_dma` is additionally `static` in `8250_port.c`, so it cannot be
called from another translation unit at all — the shim function that wraps
it (`serial8250_irq_rs_handle_rx_dma`) necessarily lives in `8250_port.c`
itself, alongside the other shims.

### Why `struct uart_port`/`struct uart_8250_port` stay opaque

Same reasoning as the immediately prior slice (neither struct has bindgen
bindings anywhere in this tree; both are large and deeply nested). `port`
stays an opaque `*mut c_void` in `8250_irq_rs.rs`. Every field read and
every subsystem call the C original performs is exposed as a narrow,
individually-named, individually-auditable `extern "C"` shim function in
`8250_port.c` (`serial8250_irq_rs_*`, 16 functions, under a new
`CONFIG_RUST_8250_IRQ` block) — a one-for-one mechanical mirror of each line
of the C body, not a reimplementation or simplification. `UART_LSR_*`/
`UART_IER_*`/`UPSTAT_*` bit tests are deliberately evaluated in the C shim
rather than ported as Rust constants: `UPSTAT_*` is a project-internal
`upstat_t` bitflag type with no stable UAPI pin (unlike `UART_LSR_*`/
`UART_IER_*`, which are `include/uapi/linux/serial_reg.h` constants), so
evaluating every bit test in C — including the `UART_LSR_*`/`UART_IER_*`
ones, for uniformity with the `UPSTAT_*` ones in the same expressions —
means no register-flag encoding crosses the FFI boundary as a Rust
constant; only the resulting `bool`/`u16` does.

### A build bug caught before any boot testing

The first draft's module-doc comment for `8250_irq_rs.rs`'s C-side
counterpart contained the literal substring `UART_LSR_*/UPSTAT_*/UART_IER_*`
inside a C block comment. `*/` is a comment terminator regardless of
surrounding context, so this closed the comment three sentences early;
everything between the accidental close and the *intended* `*/` was then
parsed as top-level C, producing `error: unknown type name 'UPSTAT_'` and
`error: unknown type name 'only'` at build time. Caught immediately by the
first build attempt on the Rust-path worktree (before any boot), fixed by
rewording the comment to avoid the literal `*/` sequence, rebuilt clean.
Recorded here in case a future translation's comment prose runs into the
same C-comment-nesting hazard.

## NO KUNIT COVERAGE — by design, separately re-confirmed for this function

The prior slice's landing doc says explicitly that its KUnit exception "is
scoped to exactly these two functions; it does not extend to any other
function/file without being separately re-confirmed." This doc is that
re-confirmation, not an inherited pass:

- The caller holds the real port spinlock at every call site of
  `serial8250_handle_irq_locked` (`guard(uart_port_lock_irqsave)` /
  `scoped_guard(...)` upstream in `serial8250_handle_irq`/sysrq paths) —
  `lockdep_assert_held_once(&port->lock)` only makes sense checked against
  the real lock, not a fake one.
- `serial8250_rx_chars`/`serial8250_tx_chars`/`serial8250_modem_status`/
  `__stop_tx`/`handle_rx_dma`/`serial8250_clear_and_reinit_fifos` (all
  still C, unmodified, called via the shim) touch real UART hardware
  registers, real DMA channel state, and — via `pm_wakeup_event` — the real
  power-management wakeup subsystem. None of this is meaningfully fakeable
  without reimplementing genuine subsystem behavior inside the test, which
  would verify the fake against itself, not the driver against anything
  real — the same class of blocker `docs/8250-tier-c-blocker-2026-07-18.md`
  identified, now accepted as a scoped, documented trade-off for this
  function specifically, per Dan's explicit instruction for this task.

## The gate that replaces KUnit here: byte-for-byte transcript comparison, 8 boots

Two isolated kernel worktrees were created via `linux_riscv_worktree.py`
(same tool, same reason, as the prior slice — the live console driver, and
another agent may be using the shared tree concurrently):
`linux-riscv-worktrees/8250-tier-c-irq` (C path, `CONFIG_RUST_8250_IRQ`
unset) and `8250-tier-c-irq-rust` (`CONFIG_RUST_8250_IRQ=y`, wiring the Rust
path into the live IRQ dispatch). Both worktrees carry
`CONFIG_RUST=y`/`CONFIG_RUST_8250_STARTUP=y` (the two already-landed,
already-verified prior slices) unchanged from the shared tree's current
state — only the new option differs between them, so the comparison isolates
exactly this slice's effect.

**Why a new Kconfig option, not `CONFIG_RUST` or `CONFIG_RUST_8250_STARTUP`:**
same reasoning as the prior slice, one level further: toggling either of
the existing options would also revert already-verified translations,
comparing the wrong thing. `CONFIG_RUST_8250_IRQ` (new, `depends on
SERIAL_8250 && RUST`, default `n`) is independently toggleable, added to
`drivers/tty/serial/8250/Kconfig` immediately after `RUST_8250_STARTUP`'s
entry.

`serial8250_handle_irq_locked` itself — the `EXPORT_SYMBOL_NS_GPL`
(namespace `"SERIAL_8250"`) function that's the actual call target from
`serial8250_handle_irq` and the sysrq-aware unlock paths — is wrapped
`#ifdef CONFIG_RUST_8250_IRQ` / `#else` in `8250_port.c`, same pattern as
the prior slice. The `#else` arm is the untouched original C body, kept
verbatim. Note the export macro here is `EXPORT_SYMBOL_NS_GPL`, not the
plain `EXPORT_SYMBOL_GPL` the prior slice's two functions used — preserved
exactly as-is on both sides of the `#ifdef`.

Each image was booted **4 times** with distinct `--run-id`s
(`tierc-irq-c-{1,2,3,4}`, `tierc-irq-rust-{1,2,3,4}`) — 8 boots total, more
than the prior slice's 6, per the task's explicit instruction given the
elevated risk class. All archived under `docs/status/boot-logs/`. All 8
boots passed independently: 17/17 KUnit suites, `ORACLE PASS`, `INIT
REACHED`, zero `not ok` lines.

### Diff methodology

Per-line `TS_PREFIX_RE` (from `scripts/kunit_oracle.py`:
`r"(?:\d{5}\.\d{3} )?"` — a bare `NNNNN.NNN` prefix, not bracketed)
stripped from every log before comparison. All 28 pairwise combinations (8
choose 2) were diffed: 6 within C-path, 6 within Rust-path, 16 cross
C-vs-Rust.

### Result: clean

**From the `KTAP version 1` line through the end of every transcript — all
17 KUnit suite results, the `# Totals` line, and the `linux-rs: initramfs
init reached, PID 1 alive` marker — all 8 logs are byte-for-byte
identical**, 362 lines each, zero deviation, across all 28 pairwise
comparisons including every one of the 16 C-vs-Rust cross-comparisons. This
is the region that reflects what happens *after*
`serial8250_handle_irq_locked` has already been exercised many times during
boot (every character the KUnit suites and console output print goes
through this function), so it is a strong signal that the two IRQ-dispatch
paths produce indistinguishable downstream behavior across repeated live
interrupt-driven console I/O, not just at one snapshot in time.

Before that point (early boot), the diffs fall into the same categories the
prior slice documented, each confirmed benign the same way (present at the
same rate within-variant as across-variant):

1. **Line-order jitter** (e.g. `Freeing initrd memory: 108K` reordering
   relative to neighboring lines) — pure QEMU early-boot interleaving
   timing, not a function of which binary is running.
2. **`cpu0: scalar unaligned word access speed is N.NNx byte access
   speed`** — a runtime-measured micro-benchmark value, varies
   within-variant as much as across-variant.
3. **Two artifacts expected from comparing genuinely different binaries**:
   the `-kernel <path>` line (each build lives in its own worktree path)
   and the `Linux version ... <build timestamp>` banner line. Unlike the
   prior slice, the `Memory: ... kernel code` line in the boot banner
   showed the *same* rounded value (`1772K`) for both variants this time —
   checked directly with `size` on both `vmlinux` binaries rather than
   relying on the boot-log's KB-rounded report: `.text` differs by exactly
   584 bytes (2020048 vs. 2019464), a real, deterministic difference below
   the log line's 1KB rounding granularity, stable across all 8 boots
   (never varies within a variant, only across).

No diff touches anything downstream of `KTAP version 1`, no diff drops,
corrupts, or reorders any line carrying real information (KUnit results,
error messages, the init-reached marker). This diff is clean, so the live
wiring stands.

## What this does and doesn't establish

**Does:** a third Tier C slice is now live-wired, and — because this
function runs on every interrupt rather than only at bring-up/teardown — a
much larger fraction of actual boot-time execution now flows through
translated Rust control flow than after the prior slice. The 8-boot,
28-pairwise-combination comparison (up from 6 boots/15 combinations) is the
most repeat-tested any Tier C slice has been to date, matching the task's
explicit instruction that this function deserved more caution, not less.

**Doesn't:** issue #25 stays open. `serial8250_do_set_termios` (explicitly
deferred, same real-lock/register entanglement) and `autoconfig`
(order/timing-fragile probing, the worst-ranked candidate per
`docs/8250-tier-c-blocker-2026-07-18.md`) remain entirely untouched, in C,
for future slices of this same standing issue. This task's KUnit exception
is scoped to exactly `serial8250_handle_irq_locked`; per the same discipline
this doc itself follows relative to the prior slice, it is not a precedent
that any other function gets to skip KUnit without its own explicit
re-confirmation.
