# 8250 Tier C: startup/shutdown control flow — blocked, not attempted

Status: **stopped before writing code** in THIS pass, per
[awto-au/linux-rs#25](https://github.com/awto-au/linux-rs/issues/25)'s own
explicit license to do so ("stopping and reporting back... is a legitimate,
valued outcome"). No changes to `linux-riscv/drivers/tty/serial/8250/*` in

**Update (2026-07-18, later same day):** the "not independently testable"
finding below is real and correct for KUnit specifically — but Dan pointed
out it's too narrow a conclusion. The functions this doc rules out for a
KUnit fake are NOT untestable in general: a real QEMU boot on the virt

against real (emulated) interrupt-controller hardware — every existing

actual subsystem, actually working, on every single `dev.py check` run.

version directly into the live path (skipping the impossible KUnit-fake
requirement) while KEEPING the boot-transcript-comparison gate from the
original scoping doc (two real boots, C vs Rust, byte-diffed, repeated) as
the actual verification — a real integration test, not a unit test, but a
real one. A follow-up agent is (re-)attempting this slice under that
scope; see issue #25's later comments for the outcome.

## What was evaluated

Per the issue and `docs/serial-8250-translation-scoping-2026-07-18.md`,
Tier C is four candidate functions in
`linux-riscv/drivers/tty/serial/8250/8250_port.c`:

| Function | Lines | Role |
|---|---:|---|
| `serial8250_do_startup` | 2271-2369 | device bring-up |
| `serial8250_do_shutdown` | 2379-2434 | device teardown |
| `serial8250_do_set_termios` | 2795-2836 | baud/LCR/FCR/IER reconfig |
| `serial8250_handle_irq_locked` | 1794-1841 | live IRQ RX/TX dispatch |
| `autoconfig` | 1070-1215 | hardware type probing |

All five (four named in the issue, `autoconfig` included per its explicit
mention) were read in full before any code was written, per the task's
instruction not to default to startup/shutdown without checking.

## Judgment: startup/shutdown is the least-bad of the four, and still blocked

Ranking by tractability, evidence-based:

1. **`autoconfig`** — worst candidate. Order-dependent, timing-sensitive
   register read-back probing (loopback test via `UART_MCR_LOOP`, IIR
   FIFO-enable-bit dispatch to `autoconfig_8250`/`autoconfig_16550a`,
   `rsa_autoconfig`/`rsa_reset`). The function's own comments concede
   fragility ("This *should* be safe... we hope!"). It also *produces*
   `port->type`, which `do_startup` then consumes — translating it first
   means translating the least well-specified function in the family, with
   downstream functions depending on getting it exactly right.
2. **`serial8250_do_set_termios`** — same real-lock/register entanglement
   as startup/shutdown (see below), explicitly deferred by the issue's own
   text as its own follow-up scope.
3. **`serial8250_handle_irq_locked`** — short (~48 lines) but runs on
   *every* live interrupt today, `lockdep_assert_held_once(&port->lock)`
   on the real port lock, zero test margin — a bug here is live on the
   console path the moment it's wired, with no graceful degradation.
4. **`serial8250_do_startup`/`serial8250_do_shutdown`** — the issue's
   suggested default, confirmed the most tractable of the four. Still
   blocked (below).

## The blocker

`serial8250_do_startup`/`serial8250_do_shutdown` are irreducibly bound to
real kernel subsystem calls with no meaningful KUnit fake — this is a
different problem than Tier B's `mem_serial_in`/`mem_serial_out`, where a
plain `[u8; N]` buffer behaves identically to real MMIO for byte-shuffling
correctness. Specifically:

- `serial8250_do_startup` calls `up->ops->setup_irq(up)`, which for this
  project's actual driver (`univ8250_driver_ops` in `8250_core.c:312`)
  resolves to `univ8250_setup_irq` -> `serial_link_irq_chain` (line 154) ->
  `request_irq(up->port.irq, serial8250_interrupt, ...)` against a live IRQ
  line, plus shared-IRQ bookkeeping (`irq_info` hash table, its own
  spinlock, `hash_for_each_possible`).
- `serial8250_do_shutdown` calls `synchronize_irq(port->irq)` — blocks
  until any in-flight interrupt handler completes; meaningless without a
  real interrupt controller — then `up->ops->release_irq(up)` ->
  `univ8250_release_irq` -> `timer_delete_sync` + `serial_unlink_irq_chain`
  -> `free_irq`.
- Both wrap their register-touching sections in
  `guard(uart_port_lock_irqsave)(port)` / `scoped_guard(...)` — real
  spinlock acquisition with IRQs disabled, not a value that can be faked
  without either a real lock (defeating the point of a unit test) or a
  stand-in whose correctness is unverifiable against the real
  `uart_port.lock` semantics.

A KUnit fake can stand in for *data* (a register file). It cannot stand in
for *"the interrupt subsystem accepted this registration"* or *"no
interrupt is currently in flight"* — faking those would mean reimplementing
genuine IRQ-core behavior (request_irq's chain-walking, shared-IRQ
book-keeping, synchronize_irq's completion-wait semantics) inside the test,
which would verify the fake against itself, not the driver against
anything real. This is exactly the interrupt-context/locking-semantics
condition the task named as a legitimate stop trigger.

**Not a scope-down opportunity either.** The purely register-I/O helpers
`do_startup`/`do_shutdown` call — `serial8250_clear_fifos` (line 494),
`serial8250_clear_interrupts` (line 705) — are already `EXPORT_SYMBOL_GPL`,
shared across many other call sites in the file (the IRQ handler,
`autoconfig`, others), not private to startup/shutdown. Translating just
those would not be "the startup/shutdown slice" the issue scoped; it would
be a different, smaller, Tier-B-shaped TU (more register accessors), and
would leave the actual control-flow function — the thing Tier C is about —
untouched. It's a legitimate future TU, but a different one, and doesn't
resolve this blocker.

## What would need to exist before this can proceed safely

- A KUnit-reachable fake for the subset of IRQ-core behavior that
  `serial_link_irq_chain`/`serial_unlink_irq_chain`/`synchronize_irq`
  actually rely on — not a full reimplementation of `request_irq`, but
  something that lets a test assert "startup registered exactly one
  handler for this IRQ, in the right shared/non-shared mode" and "shutdown
  unregistered it and waited for in-flight completion" without needing a
  real interrupt controller. No such harness exists in this project today
  (`rust/kernel/` has no IRQ-registration test scaffolding, confirmed by
  grepping for `request_irq`/`irq` fakes/mocks in `rust/kernel/*.rs` and
  finding none).
- Alternatively, a design decision to accept a narrower verification bar
  for this specific TU than the project's KUnit-suite gate has required so
  far — e.g. control-flow-only verification (assert the *sequence* of
  register writes and *which* branch is taken for a given input state,
  without touching real `request_irq`) with the IRQ-registration calls
  left as an explicit seam/stub boundary. This is a legitimate path but is
  a scoping/design call for whoever owns this issue's risk tolerance, not
  something to decide unilaterally mid-task.
- Real hardware or a QEMU-external IRQ-injection harness, if the goal is
  to verify actual interrupt-context correctness rather than just
  call-sequencing.

None of these exist today. Building the first (a proper IRQ-registration
fake) is itself a nontrivial harness project, not a same-pass addition.

## What this doesn't affect

Tier A (wired live) and Tier B (`8250_io_rs.rs`, compiled in, KUnit-verified,
not wired) are untouched and remain exactly as landed. No files under
`linux-riscv/` were modified in this pass. `dev.py check`'s current
17/17-suite baseline was not re-run since nothing changed that could affect
it.

## Recommendation for the next attempt at this issue

Don't retry `serial8250_do_startup`/`_do_shutdown` (or the other three
candidates — all rank worse) without first landing IRQ-registration test
scaffolding, or without an explicit decision to accept narrower
(control-flow/sequencing-only) verification for this specific TU. Either
is a legitimate next step; picking one is a scoping decision for whoever
next picks up issue #25, ideally made with Dan's input given the elevated
risk class this issue itself calls out.
