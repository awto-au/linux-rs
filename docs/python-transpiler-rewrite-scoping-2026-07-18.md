# Scoping: rewrite c2rust-transpile in Python? — recommend against, in favor of a narrower Python rules/post-processing layer

Status: **investigated, real recommendation below.** This is a read-only
scoping investigation (no code written, nothing in the `awtoau/c2rust`
checkout modified) into a proposal to rewrite the `c2rust-transpile` Rust
crate (~31,600 lines, the AST→Rust codegen core of this project's c2rust
fork) as a Python implementation, dropping every other crate in the
c2rust workspace this project doesn't use. The C++ AST exporter
(`c2rust-ast-exporter`) stays as-is regardless of outcome — it is out of
scope for any variant considered here.

**Bottom line up front:** the full rewrite is not recommended. The
motivating pain point — `--enable-rule` kernel-idiom additions requiring
a Rust code change, rebuild, and full regression cycle — is real but
measured at **~1% of crate scope per fix** (377 lines across six shipped
issues, sub-hour wall-clock each) and is **not actually fixed by a Python
rewrite of the whole crate**, because the current rule *dispatch*
mechanism is already thin (139 lines of infrastructure) and the
`rulesdb/rules/*.toml` files are prose/documentation, not machine-readable
match/emit specs — a Python rewrite would still require hand-written
Python recognizer functions per rule, just in a different language. The
part of today's friction that genuinely looks automatable — the
`unsafe{}`-wrapping / feature-gate-stripping / `.init_array`-trick classes
of gap documented in `docs/combined-boot-attempt-2026-07-18.md` — is a
**mechanical post-processing problem over already-emitted Rust text**, not
a reason to reimplement AST→Rust codegen, a CBOR/CFFI interface boundary,
and a Ramsey/Emscripten-style relooper from scratch. See §5 for the full
alternatives comparison and recommendation.

## 1. The exact interface boundary

The interface between `c2rust-ast-exporter` and `c2rust-transpile` is
**not a file, not a subprocess, and not JSON** — it is an in-process CFFI
call returning raw CBOR bytes in memory, decoded by a small, separate
Rust crate (`c2rust-ast-exporter`'s own `src/lib.rs` + `src/clang_ast.rs`)
that lives *between* the C++ exporter and the Rust `c2rust-transpile`
crate. This is a materially more complex boundary than "read a CBOR file"
and matters directly for the rewrite proposal, because a Python
implementation cannot simply "consume the same data" without either (a)
also reimplementing this FFI-and-decode shim in Python (via `cffi`/`ctypes`
against the same C ABI) or (b) inserting a serialization step that
doesn't exist today.

**Encode side** (`c2rust-ast-exporter/src/AstExporter.cpp`, 3477 lines):
uses `tinycbor` (`#include <tinycbor/cbor.h>`, line 46) directly against
Clang's AST via `RecursiveASTVisitor`. Each AST/type node is encoded as a
CBOR array via a shared `encodeNode`-style helper (`AstExporter.cpp:180-221`)
carrying: entry ID, a `uintptr_t` tag, then per-kind extras. The whole
per-translation-unit result is returned from `ast_exporter()` (a C ABI
function declared in `AstExporter.hpp`) as a `struct ExportResult`
(`names`/`bytes`/`sizes`/`preprocessed` parallel arrays, one entry per
output file) — never written to disk in production use (a debug dump path
exists in commented-out code in `c2rust-ast-exporter/src/lib.rs:46-49`).

**FFI + decode side** (`c2rust-ast-exporter/src/lib.rs`, 153 lines, and
`src/clang_ast.rs`, 305 lines — a *separate* crate from `c2rust-transpile`
itself): `get_ast_cbors()` (`lib.rs:61-93`) calls the C++-linked
`ast_exporter()` FFI function under a global `CLANG_MUTEX` (libClangTooling
is documented not thread-safe), marshals the returned C struct into a
`HashMap<String, (Vec<u8>, Option<String>)>` (`marshal_result`,
`lib.rs:121-153`), then `get_untyped_ast()` (`lib.rs:33-56`) runs
`serde_cbor::from_slice` on the raw bytes and calls `clang_ast::process()`
(`clang_ast.rs:181-305`) to build a `clang_ast::AstContext` — a
loosely-typed intermediate (`HashMap<u64, AstNode>`, `HashMap<u64,
TypeNode>`, both still carrying `Vec<serde_cbor::Value>` "extras" fields
that haven't been destructured into named fields yet). Node/type *tags*
are decoded via `unsafe { std::mem::transmute::<u32, ASTEntryTag> }`
(`clang_ast.rs:169-179`) against `enum ASTEntryTag`/`enum TypeTag`
generated by `bindgen` from the **same shared C++ header**
(`c2rust-ast-exporter/src/ast_tags.hpp`) the C++ encoder uses — i.e. tag
numbering is guaranteed consistent only because both sides `#include`/
`bindgen` the identical header at build time, not by any wire-format
contract. `ast_tags.hpp` defines 15 `ASTEntryTag` decl/stmt-adjacent
values (0-14, spanning both decl and macro tags in one block; separately
17 statement tags at 100+ and ~40 expression tags at 200+/300+), 49
`TypeTag` values (400/500+), and a 6-value `StringTypeTag`.

**Where "read exported AST" ends and "generate Rust code" begins**:
`clang_ast::AstContext` (the loosely-typed CBOR-decoded form, still in the
`c2rust-ast-exporter` crate) is handed to `c2rust-transpile`'s own
`c_ast::conversion` module (`c2rust-transpile/src/c_ast/conversion.rs`,
2652 lines), which re-IDs every node (`IdMapper`, `conversion.rs:56-90` —
Clang IDs are not 1:1 with the internal ID space because some Clang nodes
get inlined/collapsed and others get synthesized) and converts into the
crate's own strongly-typed `TypedAstContext`
(`c2rust-transpile/src/c_ast/mod.rs`, 3525 lines — the real target schema,
enumerated in full in §2). Only past this point does anything resembling
Rust-code generation begin (`translator/mod.rs` and siblings). **A Python
rewrite's realistic interface boundary is therefore either: reimplement
the CFFI-and-CBOR-decode shim in Python and consume `clang_ast::AstContext`
directly (harder, touches unsafe C ABI/tag-transmute code), or keep the
existing Rust `c2rust-ast-exporter` crate as a small helper binary that
serializes `TypedAstContext` (post-`conversion.rs`) to a file (e.g. JSON)
for Python to read (simpler, but adds a new serialization boundary and a
small amount of new Rust code that doesn't exist today — the crate
currently never touches disk for this data).** Either way, "the Python
side would consume the same data the current Rust `c2rust-transpile`
crate consumes from the exporter" undersells the work: there is no
existing serialized/documented wire format at that exact boundary today,
only an in-process Rust struct.

## 3. CFG relooper and Rust-type-system risk assessment

**Algorithm identity, confirmed from source, not inferred**: `cfg/relooper.rs`'s
own module doc (lines 1-12) explicitly cites it as an implementation of
the classic **Ramsey/Kripken "Relooper" algorithm from the Emscripten
paper** (linking the actual ACM DOI), "with the addition of some
heuristics and Rust-specific control-flow constructs" — not a novel or
simplified derivative. This is confirmed independently non-trivial by its
own history: `awtoau/c2rust`'s upstream ancestor shows a **full rewrite**
of this exact module (`6c103071f`, "Rework relooper algorithm for fun and
profit", +992/-changed lines in `relooper.rs` alone, +796 in
`structures.rs`, plus a dedicated snapshot-test suite of hand-crafted
control-flow shapes: `gotos.c`, `if_else.c`, `loops.c`, `irreducible.c`,
`switch.c`) — evidence that even the original, experienced Rust
maintainers found this code hard enough to need a wholesale redo once
already.

**Where the real complexity and bug risk actually live**: `make_loop()`
(`relooper.rs:417-531`) does dominator-based node partitioning between
loop-body and follow-blocks; `compute_dominators()` (`relooper.rs:762-805`)
is a textbook iterative fixpoint algorithm (the single most
classically error-prone pattern in compiler literature — it has its own
8-test unit suite, lines 831-955, specifically because of this);
`transitive_closure()` is a worklist/DFS fixpoint recomputed on every
recursive `relooper()` call; `simplify_structure()`
(`relooper.rs:535-700`) is a second, order-dependent recursive pass
merging `Multiple`/`Simple` structures. This is genuinely some of the
gnarliest code in the crate, and none of it is uniquely un-portable to
Python — but a from-scratch Python reimplementation would be
reimplementing a published, subtle, already-once-rewritten graph
algorithm with a materially thinner safety net (no borrow checker, no
`IndexMap`/`IndexSet` type discipline forcing key-shape consistency).

**Does Rust's exhaustiveness checking actually matter here, empirically?**
Investigated via git history of every real bug in `cfg/` (both upstream
and this project's own fixes from 2026-07-17/18) plus every one of
today's traced issues (#13, #14, #16-#20). The answer is **mixed, and
mostly not in the direction the proposal's hypothesis assumed**:

- Two of four real historical `cfg/` bugs (`513d47eea` upstream,
  `d1cfda9e1` from this project — the **dominant crash cause in the
  linux-rs corpus, 48/71 crash files**, per its own commit message and
  issue #6) are **unchecked map/graph-lookup assumptions**
  (`self.nodes[&lbl]` panicking on an absent key) — a bug class Rust's
  type system does **not** prevent (`map[&k]` panics identically to
  Python `dict[k]` raising `KeyError`; this is a defensive-coding
  discipline question, not a compile-time-checking one, and is equally
  hard in both languages).
- A third (`2d556ecb3`) is a case where a match **was** exhaustive at the
  top level (`CStmtKind`, correctly dispatching to the `Attributed` arm)
  but a **nested** match inside that arm used a wildcard `_ =>
  panic!(...)` instead of naming all cases — i.e. Rust's exhaustiveness
  checking provided **zero** protection here, because the code opted out
  of it with a wildcard. This is exactly the failure mode Python would
  also have.
- Of the six real, fixed issues from today (#13, #14, #16-#19 — #20 has
  no fix, see §4), **three (#13, #16, #17) are "missing visitor override"
  bugs** — but they live in the **C++ AST exporter**
  (`AstExporter.cpp`'s `RecursiveASTVisitor`-based `Visit*`/`Traverse*`
  methods), not in any Rust `match`. C++'s visitor-override pattern
  provides **no** compile-time exhaustiveness guarantee either (a missing
  `VisitFooDecl()` silently no-ops) — meaning this exact bug class already
  exists today in a statically-typed, non-Rust part of the pipeline that
  is explicitly **out of scope** for the Python rewrite proposal (the
  exporter stays as-is). This undercuts the proposal's own risk framing:
  the part of the pipeline that already exhibits "missing case, no
  compiler help" bugs isn't the part being proposed for rewrite.
- The crate's own highest-arity, most exhaustiveness-sensitive enum
  (`CTypeKind`, 60 variants) already **opts out** of exhaustive matching
  at its primary use site (`convert_type.rs:210-335`, ~35 named arms
  terminating in a wildcard `Err(format_err!("Unsupported type
  {:?}", t))`) — so for the single enum where Python's lack of
  exhaustiveness checking would matter most in theory, the existing Rust
  code already behaves like Python would (silent-ish runtime fallback,
  not a compile error) at exactly that spot.
- Where a genuinely exhaustive match **does** exist and **would** catch a
  missing variant at compile time (`translator/mod.rs:3306`'s
  `convert_expr()` over all 29 `CExprKind` variants, no wildcard arm;
  `cfg/mod.rs:1432`'s `convert_stmt_help()` over all 19 `CStmtKind`
  variants, no wildcard) — that protection is real and would be lost in a
  literal Python port using `if/elif` chains. Across the crate, roughly
  103 `match ...\.kind` sites reference these AST enums; a rough
  correlate (raw variant-name occurrence counts) puts `CTypeKind` uses at
  320, `CExprKind` at 181, `CDeclKind` at 120, `CStmtKind` at 67 — a
  non-trivial surface a Python rewrite would need matching test coverage
  or a hand-built exhaustiveness linter to protect at all.

**Net assessment**: the CFG/relooper module and the AST-variant match
surface are real, evidenced risk if ported to Python naively — but the
project's actual historical bug pattern (unchecked lookups, wildcard
arms that already defeat exhaustiveness checking, and missing-visitor
bugs in the C++ layer that's staying regardless) is **not** the failure
mode Rust's `match` exhaustiveness uniquely prevents. A Python rewrite
would need to deliberately build its own exhaustiveness discipline (e.g.
`enum.Enum` + a lint step failing CI on any non-exhaustive `match`-style
dispatch over the AST kind enums) to not regress below the current bar —
achievable, but real, additional design work, not a free byproduct of
"Python is simpler."

## 4. Pluggable rule architecture — the proposal's actual core motivation

The coordinator confirmed this is the primary motivation for the whole
proposal, so it's assessed directly rather than folded into general risk.

**How `--enable-rule` actually works today, precisely** (traced from
`c2rust-transpile/src/kernel_idioms.rs`, 139 lines total): `KernelIdiomRule`
is a plain Rust enum (`WarnOn`, `FlsFamily`, `SwapMemSwap`,
`AddrLabelPlaceholder`, `ExportSymbol`, plus an `All` catch-all) with
`strum`-derived kebab-case string parsing for the CLI. `KernelIdiomRules`
is just `HashSet<KernelIdiomRule>` with an `is_enabled()` gate. **There is
no registry, no trait object, no dispatch table** — each rule's actual
recognizer-and-emitter logic is a bespoke method living inline in
whichever `translator/*.rs` file already handles the relevant AST node
kind (e.g. `bit_scan_idiom_body()` in `translator/functions.rs:422-434`
for `FlsFamily`, `warn_on_condition()` in `translator/mod.rs:4174-4178`
for `WarnOn`), gated by a one-line `if !...is_enabled(...) { return None
}` check at the top. Total machinery — infrastructure plus all five
rules' dispatch-site glue — is under ~300 lines; the "rule" itself is
just wherever in the 16,349-line translator subsystem the relevant AST
shape is already being visited.

**Is `rulesdb/rules/*.toml` a machine-readable rule spec today?** No —
checked directly across all 30 files. The schema (`[match]` with a `c`
field, `[emit]` with a `rust` field, `[provenance]`, `[validation]`,
etc.) is **prose**: `[match].c` for rule 0006 (fls-family) is the string
`"fls(x) /* 1-based, fls(0)==0 */; __fls(x) /* 0-based, UB at 0 */;
fls64/__ffs64 variants"` — a human-readable description of what to
recognize, not a parseable pattern/AST-shape spec. `[emit].rust` is
similarly prose describing the target transformation, including inline
warnings like "NOT A RENAME... `fls(x) -> x.leading_zeros()` with no
further change is WRONG." Only 2 of 30 rule files even contain a
`[match.<subkey>]` table structure suggesting anything beyond flat
prose fields, and neither is an executable matcher. **Every one of the
30 documented rules requires a human (or agent) to read the TOML and
hand-write the corresponding Rust recognizer/emitter** — exactly as
happened for the 5 rules actually implemented so far. Of the 30 rules,
20 are tier 1 (mechanical, well-specified), 5 tier 2, 5 tier 3
(explicitly region/judgement-based, e.g. `safe-lift-lock-guard`,
`lkmm-primitive-shim` — not realistically automatable as pattern-match
rules regardless of implementation language).

**Real cost evidence for adding a rule today** (from git history of the
five shipped rules, `2e819e339`/`f845699ba`/`91b0d41be`/`bf319b94b`/`122dc1361`):
diffs ranged 12-120 lines across 1-3 files each; a full-workspace
incremental release rebuild after touching `c2rust-transpile` measured
**26.3 seconds** (`cargo build --release -p c2rust-transpile -p c2rust`,
timed directly, not estimated). The friction isn't compile time — it's
that `run_c2rust_baseline.py`'s full-corpus regression check is the real
gate (per `build_c2rust.py`'s own documented incident: a stale-binary
regression from `--bin c2rust`-only builds silently re-tested an old
transpile binary and regressed 7→317 dropped_decls before being caught),
and that gate exists to catch genuine correctness regressions across the
whole kernel-source corpus, not to gatekeep the code-editing step itself.

**Would Python's dynamic nature genuinely help?** Partially, but less
than the proposal's framing suggests, because the current friction is
not "Rust's `match` is hard to add an arm to" — adding a `KernelIdiomRule`
variant plus a recognizer function is already a small, additive, isolated
change (median ~70 lines, confirmed above), architecturally no harder
than adding a Python function and registering it via a decorator. A
Python rewrite genuinely *could* offer:
- **Hot-reload / no-rebuild iteration** — real win; 26s is not
  large but is not zero, and a Python edit-run loop is faster in the
  small.
- **Rules as ordinary functions with decorator registration** — a real,
  modest ergonomics improvement over "find the right `translator/*.rs`
  file and thread through an `is_enabled()` check," but not a
  categorically different shape from what exists (both are "write a
  recognizer function, gate it, call it from the right AST-visit site").
- **Rules as data read directly from `rulesdb/rules/*.toml`'s existing
  schema "with no schema translation step"** — **this specific framing
  does not hold up**: today's TOML schema is prose by design (it exists
  to document intent, matching criteria in English, and validation
  provenance for humans/agents, per its own `[validation].negative`
  fields warning about specific wrong-translation pitfalls like the
  fls-family off-by-`BITS` bug). Making it *executable* would mean
  designing an entirely new, actually-machine-readable match/emit DSL —
  real, non-trivial design work belonging to *either* language choice
  equally, not something Python's dynamism grants for free. The project
  has, in fact, already built exactly this kind of TOML-reading Python
  tool once — `check_c2rust_rule_conformance.py` — but as a **read-only,
  regex-based conformance checker** over already-emitted `.rs` text, not
  as a codegen driver; it explicitly documents "text/regex scanning...
  not a real Rust AST parser" and flags tier-3 rules as
  `not_mechanically_checkable`. This existing tool is strong evidence for
  what a Python "rules layer" realistically looks like in practice: a
  post-processing/verification pass over text, not an AST-driving
  codegen core — which points toward alternative (b) in §5, not (a).

**Conclusion for this section**: a genuinely pluggable, hot-reloadable,
decorator-registered rule system is a real and buildable idea, but it
does not require rewriting the 31,600-line AST→Rust codegen core, the CFG
relooper, or the CBOR/CFFI interface boundary to get it. It requires (i)
a real match/emit DSL or Python-function-registration layer sitting at
the same "already-visiting-this-AST-node" call sites the Rust code uses
today, which could be built as a **plugin hook inside the existing Rust
transpiler** (e.g. embed a scripting layer, or expose an FFI rule-callback
API) at far less cost than a full-language rewrite, or (ii) — the
option this doc recommends, see §5(b) — move rule application to a
post-processing pass over emitted Rust text/AST, which sidesteps needing
any change to the Rust core's AST-visiting internals at all and is
exactly the shape `check_c2rust_rule_conformance.py` already proves out
for verification.

## 5. Effort estimate, grounded in today's real fix data

Investigated all seven issue numbers referenced (#13, #14, #16-#20) via
`gh issue view` against `awtoau/c2rust` (all seven exist there; the
`linux-rs#N` numbers seen elsewhere are a distinct, unrelated local
numbering series).

| Issue | Root cause class | Fix size | Open→close |

**Aggregate**: 377 net lines changed across six shipped, independently
diagnosed fixes ≈ **1.16% of the crate's 32,473-line real size** (all
`.rs` files including blanks/comments — the ~31,600 figure used
elsewhere in this doc is source-only). Every fix was additive/narrowly
localized (1-7 files, 12-120 lines); none touched more than 7 files.
Every fix's open→close window was under one hour, and **each issue was
filed with the root cause already traced** (debug-instrumented
reproduction and exact source citations in hand at filing time) — the
measured time is "fix + rebuild + full-corpus regression baseline +
merge," not "discover the bug from scratch."

**What this data does and doesn't support**: it is real, direct evidence
that *narrow, pre-diagnosed, additive bug fixes within the existing
architecture* are cheap — on the order of under an hour and under 1% of
crate scope each. It is **not** evidence about the cost of a from-scratch
rewrite of the whole architecture in a different language, which has no
existing skeleton to extend and would need to reproduce: the CFFI/CBOR
interface shim (§1), the 13/29/19/60-variant AST conversion layer (§2),
the Ramsey/Emscripten relooper (§3, already needed one full rewrite by
its original Rust authors), and ~16,349 lines of expression/statement/
declaration translation logic covering hundreds of match sites — while
also building new exhaustiveness discipline to not regress below what
Rust's `match` already catches for free at specific sites (§3). Treating
the ~1% narrow-fix ratio as if it scaled to "~1% of a rewrite's effort"
would be an unsupported extrapolation — flagging this explicitly rather
than fabricating a week/month estimate the data doesn't support.

**Directionally**, the closest actual data point for full-scope risk is
issue #20 — the *one* issue in the set that isn't a narrow, localized bug
fix but a broader "does the whole output actually work in a real build"
integration finding, and it remains **open, unfixed**, sourced from a
27KB narrative investigation doc (`docs/combined-boot-attempt-2026-07-18.md`)
that took real, substantial exploration to characterize even without
attempting a fix. A full Python rewrite is architecturally much closer in
kind to "re-derive #20's entire pipeline from scratch and get it back to
today's baseline" than to "six more #13-style patches" — i.e. the
realistic order of magnitude is **multi-session, not incremental**, even
without a fabricated week count.

## 6. Alternatives comparison and recommendation

**(a) Full Python rewrite of `c2rust-transpile`, as proposed.**
Reproduces the CFFI/CBOR boundary (§1), the 60-variant `CTypeKind`/
29-variant `CExprKind` conversion and dispatch surface (§2), and the
already-once-rewritten relooper (§3) from scratch, in a language with no
compile-time exhaustiveness checking by default (real risk, though
narrower than the proposal's framing — §3's synthesis). Does not, by
itself, solve the pluggable-rules motivation any better than option (c)
below (§4's conclusion) — the actual rule-dispatch machinery is already
thin; the bulk of the crate being proposed for rewrite is the AST
conversion and relooper, which are exactly the highest-risk, most
proven-hard parts, and the parts LEAST related to "add a kernel-idiom
rule faster." **Not recommended.**

**(b) A narrower Python post-processing layer over c2rust's raw Rust
output**, mechanically automating the gap classes already documented and
hand-fixed for real in `docs/combined-boot-attempt-2026-07-18.md`: strip
disallowed unstable-feature crate attributes, wrap known-unsafe operation
shapes in explicit `unsafe {}` blocks, strip the dead `.init_array`/
`__UNIQUE_ID_addressable_*` constructor trick, redirect stray `libc` calls
to TU-local `extern "C"` declarations, pair `#[export]` removal with
`#[no_mangle]` insertion, and delete unused register-variable pseudo-
globals. This doc found **7 distinct, mechanically-characterizable gap
classes**, several explicitly confirmed **recurring/corpus-wide** across
independent files (`docs/combined-boot-attempt-2026-07-18.md`'s "same
class as #6/#7" and "confirmed present again" language), each with a
concrete, bounded fix pattern already worked out by hand at least twice.
This is a small, immediately-payoff-bearing project: it operates purely
on text/AST of already-emitted `.rs` (no CBOR/CFFI boundary, no relooper,
no C AST conversion needed at all), and the project has already proven
the "Python reads `rulesdb/rules/*.toml` and reasons about c2rust output"
pattern works in production via `check_c2rust_rule_conformance.py` (today
read-only/verification; extending it to *emit* fixes rather than only
*flag* violations is a natural, bounded next step, not a new paradigm).
**This is where this doc recommends focusing:** it directly addresses
the mechanical-gap pain that's actually blocking real boots today (issue
#20 class), it's boundable and testable against the existing regression
harness, and it does not touch the two riskiest, most proven-hard
subsystems (relooper, AST conversion) at all.

**(c) Stay with Rust/C++, trim the workspace to load-bearing crates.**
Real and available regardless of any Python decision: dropping
`c2rust-refactor` (47,463 lines), `c2rust-analyze` (28,133),
`cross-checks` (5,476), `pdg` (2,741), `dynamic_instrumentation` (2,736)
— confirmed zero-referenced by this project's own scripts — removes
**86,549 lines**, nearly 3x `c2rust-transpile`'s own size, from the
workspace this project has to reason about, build, and keep compiling,
with **zero rewrite risk** since nothing behaviorally changes. This is
lower-effort and lower-risk than either (a) or (b) and is not mutually
exclusive with either — it should happen regardless of what's decided
about (a)/(b), simply because it costs little and removes real
maintenance surface (`Cargo.lock` churn, build time, cognitive overhead
of "which crate does this belong in" for future agents) for no
functional loss.

**Recommendation: pursue (b) and (c) together; do not pursue (a).**
(c) first/immediately — it is unambiguous, low-risk, and already fully
justified by the zero-reference grep evidence in §2. (b) next, scoped
exactly to the gap classes already characterized in
`docs/combined-boot-attempt-2026-07-18.md`, extending
`check_c2rust_rule_conformance.py`'s proven TOML-reading pattern from
"detect" to "detect and mechanically fix." This gets the project a real,
working, immediately-useful Python tool that operates in the exact
territory (kernel-idiom rules, output-shape gap classes) the original
proposal's motivation cared about, without betting a multi-session,
high-risk rewrite against a crate whose hardest parts (relooper, 60-way
type dispatch, CFFI/CBOR boundary) have already needed one from-scratch
rewrite in their current, more type-safety-assisted language, and whose
actual measured historical bugs (§3) are dominated by defensive-lookup
discipline gaps that a Python rewrite would not fix for free.
