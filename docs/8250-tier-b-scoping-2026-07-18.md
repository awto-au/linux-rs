## What was translated, and why only this much

The C original (`drivers/tty/serial/8250/8250_port.c`, lines 349-417)
provides **six** `struct uart_port.serial_in`/`.serial_out` variant pairs,
selected at probe time by `set_io_from_upio()` based on `p->iotype`:
`mem_serial_in`/`mem_serial_out` (`UPIO_MEM`), `mem16_*` (`UPIO_MEM16`),
`mem32_*` (`UPIO_MEM32`), `mem32be_*` (`UPIO_MEM32BE`), `io_serial_in`/
`io_serial_out` (`UPIO_PORT`, `CONFIG_HAS_IOPORT`-gated), `hub6_*`
(`UPIO_HUB6`, also `CONFIG_HAS_IOPORT`-gated), and `no_serial_in`/
`no_serial_out` (the unrecognized-iotype fallback).

**Translated: `mem_serial_in`/`mem_serial_out` only.** Re-traced the OF
probe path the scoping doc already identified (`8250_of.c`'s
`of_platform_serial_probe` -> `of_platform_serial_setup`) to confirm it
still holds on this tree: QEMU virt's `"ns16550a"`-compatible UART matches
`of_platform_serial_table[]`, resolves `port_type = PORT_16550A`, takes the
generic path (no vendor `.setup` callback ‚Äî the `switch (type)` in
`of_platform_serial_setup` only special-cases `PORT_RT2880`/`PORT_NPCM`),
and `port->iotype` resolves to `UPIO_MEM` because the device-tree `reg`
property is a `mapbase`, not `iobase`. `set_io_from_upio()` then wires
`p->serial_in = mem_serial_in` / `p->serial_out = mem_serial_out`. That is
the only iotype this project's actual boot target has ever produced or
will produce without a board change.

**Deliberately left in C, undocumented as a gap (documented as a scoping
decision):**
- `mem16_serial_in`/`_out`, `mem32_serial_in`/`_out`,
  `mem32be_serial_in`/`_out` ‚Äî same offset/shift shape as `mem_serial_in`,
  different register width (`readw`/`writew`, `readl`/`writel`,
  `ioread32be`/`iowrite32be`). Zero real coverage: no `CONFIG_SERIAL_8250_*`
  option in this project's `.config` selects a device using any of these.
  Translating them would be low-marginal-effort (same shape, swap the
  bindgen call) but adds `unsafe` surface with no verification path this
  project can actually exercise ‚Äî there's no real or fake register file to
  test them against that isn't purely hypothetical, unlike `mem_serial_in`
  where the KUnit fake-backing test at least mirrors the exact byte-width
  and shift semantics the live device uses.
- `io_serial_in`/`io_serial_out`, `hub6_serial_in`/`hub6_serial_out` ‚Äî port
  I/O (`inb`/`outb`/`outb`), `CONFIG_HAS_IOPORT`-gated. riscv64 QEMU virt
  has no legacy ISA I/O port space wired to this driver; translating these
  would need a from-scratch KUnit mock for a bus type this project's target
  literally cannot exercise even in principle (no fake can stand in for "no
  I/O port space exists").
- `no_serial_in`/`no_serial_out` ‚Äî trivial (`~0U` / no-op) stub for an
  unrecognized iotype. Not hardware I/O, not meaningfully translatable in a
  way that teaches anything about MMIO correctness.

This mirrors the task's own explicit license to scope down: "the other
variants... can be scoped down or stubbed/left in C if they add
disproportionate risk for zero real coverage on this target." Genuinely
zero real coverage was confirmed by re-tracing the probe path, not assumed.

## MMIO primitive: `bindings::readb`/`writeb`, not `kernel::io::Mmio`

Read `rust/kernel/io.rs` in full before writing any code. It provides a
real, already-used-in-this-project abstraction (`Io`/`Mmio`/`IoCapable`
traits, `impl_mmio_io_capable!` macro) for bounds-checked, typed MMIO
register access. Did not use it here, for the same reason
`lib/iomem_copy_rs.rs` (already landed, TU 32) didn't: `Mmio` models a
**bounds-checked, Rust-owned** MMIO region, constructed via
`MmioRaw::new`/`ioremap` with a known `SIZE`. `uart_port.membase` is a bare
`unsigned char __iomem *` whose lifetime, validity, and `ioremap` call are
entirely owned by C code in `8250_of.c`'s probe path ‚Äî well outside this
TU's scope. Wrapping it in an owning `Mmio` here would require either
unsound pretend-ownership (constructing an `Mmio` around a pointer this
Rust code doesn't actually own or control the lifetime of) or a materially
larger refactor of the C-side port lifecycle to hand real ownership to
Rust. Neither is appropriate for a narrowly-scoped accessor swap.

Instead: `bindings::readb`/`bindings::writeb` directly ‚Äî already generated
and callable (`rust/bindings/bindings_helpers_generated.rs:1224-1241`,
`#[link_name = "rust_helper_readb"]` / `"rust_helper_writeb"`, backed by
`rust/helpers/io.c`'s existing `rust_helper_readb`/`rust_helper_writeb`
shims, which have been in the tree since before this TU ‚Äî `Mmio` itself
calls these same two functions internally, see `impl_mmio_io_capable!` in
`io.rs`). This is genuine `unsafe` volatile MMIO via the kernel's own
ordered accessor, not a raw pointer dereference: `readb`/`writeb` carry the
`__io_br`/`__io_ar`/`__io_bw`/`__io_aw` ordering-barrier semantics
(`include/asm-generic/io.h`) the C original relies on. No new C shim needed
‚Äî `readb`/`writeb` (unlike `iomem_copy`'s `__raw_read*`/`__raw_write*`,
which needed new `rust_helper_raw_*` shims when TU 32 landed) were already
wrapped.

Computed `offset << regshift` in Rust exactly as the C body does, then
`membase.add(shifted)`, then `readb`/`writeb` on the resulting pointer ‚Äî
same three-step shape as `mem_serial_in`/`mem_serial_out`'s C bodies, no
strengthened or weakened safety contract either direction.

## KUnit coverage against a fake register backing

The scoping doc's Tier B gate is explicit: "KUnit coverage exercising the
accessor functions against a mock/fake register backing (not real MMIO ‚Äî
no hardware side effects to fake, just byte-shuffling correctness)." No
landed TU in this project (35 TUs prior to this one) has used a real
in-kernel Rust KUnit test (`#[kunit_tests(...)] mod tests { #[test] fn }`)
before ‚Äî every prior TU was verified via host-side diff-oracle
(`bench/diff_*.{c,rs}`), which doesn't apply here (there's no meaningful
host-side reference implementation for `readb`/`writeb` against a real
`__iomem` pointer). This is the first TU in the project to add a genuine
Rust-native KUnit suite, using the same `#[kunit_tests]`/`#[test]` macro
already used internally by `rust/kernel/kunit.rs`'s own self-test
(`rust_kernel_kunit`, `CONFIG_RUST_KUNIT_SELFTEST`) and by several other
vendored `rust/kernel/*.rs` files (`bitmap.rs`, `str.rs`, `kvec.rs`,
`bitfield.rs`, `alloc/allocator.rs`) ‚Äî established upstream pattern, newly
applied to a linux-rs-authored TU.

**The fake backing is a real, ordinary kernel-owned `[u8; N]` buffer, not a
mock trait or a reimplementation.** `readb`/`writeb`'s ordering barriers
are memory-barrier instructions (`mb()`-family, arch-dependent) ‚Äî always
well-defined against plain readable/writable RAM, not just device MMIO;
the "ordering" they provide is a no-op-but-safe concern for RAM, not a
correctness hazard. So the KUnit test calls the *exact same*
`mem_serial_in_rs`/`mem_serial_out_rs` functions the (currently inert)
production path would call, pointed at a stack-local byte array's address
instead of a real device's `membase`. This exercises the genuine
`bindings::readb`/`bindings::writeb` FFI calls and the real offset/shift
arithmetic ‚Äî not a stand-in.

Four test cases, `rust_8250_mem_serial_io` suite
(`drivers/tty/serial/8250/8250_io_rs.rs`):
1. `round_trip_no_shift` ‚Äî write then read back at a fixed offset,
   `regshift=0`; confirms neighbouring bytes are untouched.
2. `regshift_scales_offset` ‚Äî `offset=2, regshift=2` must land at byte
   index 8 (`2 << 2`), not byte index 2; the property that would silently
   break if the shift were dropped or applied in the wrong order.
3. `value_truncated_to_byte` ‚Äî writing `0xDEAD_BEEF` (a `u32`, matching
   `struct uart_port.serial_out`'s signature) only stores the low byte
   (`0xEF`); confirms `writeb`'s implicit truncation, not a widened write.
4. `independent_registers_no_crosstalk` ‚Äî four distinct offsets written in
   one order, read back in another; confirms no aliasing between
   register slots, the property the real driver depends on to address
   `UART_RBR`/`UART_IER`/`UART_FCR`/`UART_LSR`/... independently within one
   `membase` region.

All four pass in-kernel (not host-simulated): `dev.py check`'s captured
boot log shows `# rust_8250_mem_serial_io: pass:4 fail:0 skip:0 total:4`,
`ok 17 rust_8250_mem_serial_io`.

## Confirmation nothing is bound to the live console path

This was the task's single most important constraint. Verified three ways:

1. **`8250_port.c` has zero diff.** `git diff` against the pre-TU tree
   shows no changes to that file at all ‚Äî `set_io_from_upio()` still reads
   `p->serial_in = mem_serial_in; p->serial_out = mem_serial_out;`
   (the C functions, lines 437-438), completely untouched.
2. **Only two files changed**: the new
   `drivers/tty/serial/8250/8250_io_rs.rs`, and an additive-only
   `drivers/tty/serial/8250/Makefile` change (`8250_base-$(CONFIG_RUST) +=
   8250_io_rs.o`, alongside the pre-existing `8250_helpers_rs.o` line ‚Äî no
   line removed, no existing rule changed).
3. **`nm vmlinux`**: `mem_serial_in_rs`/`mem_serial_out_rs` are present as
   global text symbols (confirming the code is compiled in and linked),
   but grepping `8250_port.o`'s undefined-symbol table (the technique used
   to confirm the Tier A wiring in the prior milestone doc) turns up
   nothing referencing either symbol ‚Äî because `8250_port.c` never calls
   them. Compiled in, unit-tested, unreachable ‚Äî exactly the gate's
   wording.

## Incidental fix: `dev.py boot`/`check` didn't work with worktree trees

`scripts/linux_riscv_worktree.py`'s own docstring instructs agents to use
`LINUXRS_TREE=linux-riscv-worktrees/<name>` with `dev.py`, but `dev.py
boot()` passed `boot_qemu.py --tree TREE.name` ‚Äî `Path.name` silently
strips any subdirectory prefix, so `linux-riscv-worktrees/8250-tier-b`
became just `8250-tier-b`, and `boot_qemu.py` (correctly) looked for a
kernel image at `<repo>/8250-tier-b/arch/riscv/boot/Image`, which doesn't
exist. `dev.py build` worked fine (uses `TREE` directly, a full `Path`);
only `boot`/`check` were affected. Fixed by threading the raw
`LINUXRS_TREE` env string through as `TREE_REL` and passing that instead
of `TREE.name` ‚Äî `scripts/dev.py`, two-line change. This is exactly the
scenario the worktree script's own docstring flagged as a known
coordination cost of concurrent kernel-tree work; this was a mechanical
gap in the tooling meant to support it, not a design disagreement.

## What this does and doesn't establish

**Does:** the second of three explicit tiers in the original scoping doc
is now real, in-tree, KUnit-verified Rust, with the exact "compiled in but
inert" staging that doc's own gate called for before Tier C is even
considered. Establishes the first genuine in-kernel Rust KUnit test in
this project (prior TUs used only host-side diff-oracles), a pattern
future `unsafe`/hardware-adjacent TUs can reuse.

**Doesn't:** Tier C (`serial8250_do_startup`/`_do_shutdown`,
`serial8250_do_set_termios`, the IRQ path, `autoconfig`, the console write
path itself) remains completely untouched ‚Äî explicitly out of scope for
this task, gated on Tier B succeeding first, which it now has. The much
stronger gate before any live-console swap (byte-for-byte side-by-side
transcript comparison across multiple runs, as documented in the original
scoping doc) has not been attempted and is not part of this TU's claim.
`mem16_*`/`mem32_*`/`mem32be_*`/`io_*`/`hub6_*`/`no_*` remain in C,
untranslated, with the reasoning above ‚Äî a future TU targeting a different
board (one that actually selects a non-`UPIO_MEM` iotype) should translate
whichever variant that board needs, verified against that board's own
boot, not speculatively here.
